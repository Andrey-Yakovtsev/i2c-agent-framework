# I2C Self-Diagnostics: Consistency Review

## Режимы запуска

Оркестратор явно указывает режим при вызове.

**Full mode** (по умолчанию, команда `/i2c-check`): проверяй все секции 1–7, читай все документы в `docs/`.

**Scoped mode** (`scope: "new-rfc"`, вызов из `create-rfc` Шаг 5.5): проверяй **только секции 3, 4, 8 и 9**.
- Получаешь: новый RFC + зависимые RFC (полный текст) + все ADR. Не читай PRD и другие документы.
- Пиши отчёт в `.i2c/scratch/rfc-[N]-consistency.md` (не в `consistency-report.md`).
- В summary указывай только проблемы касающиеся переданных документов.

---

## Инструкция для агента

Ты выполняешь консистентность-проверку всей документации проекта.
Прочитай все документы в `docs/` и ответь на каждый вопрос ниже.
Для каждого вопроса: ✅ OK / ⚠️ Предупреждение / ❌ Критическая проблема.

---

## 1. Наличие документов

- [ ] PRD существует (`docs/PRD.md`)
- [ ] Хотя бы один ADR существует (`docs/ADR-*.md`)
- [ ] Хотя бы один RFC существует (`docs/rfc/RFC-*.md`)
- [ ] MEMORY.md актуален (содержит решения из PRD и ADR)

## 2. PRD → ADR консистентность

- [ ] Каждое ключевое архитектурное решение из PRD покрыто ADR
- [ ] ADR-решения не противоречат scope из PRD
- [ ] Технологический стек в ADR соответствует ограничениям из PRD

## 3. ADR → RFC консистентность

- [ ] Каждый RFC использует технологии из принятых ADR
- [ ] Ни один RFC не предлагает альтернативу уже закрытому решению
- [ ] RFC покрывают все компоненты упомянутые в ADR

## 4. RFC внутренняя консистентность

- [ ] Все RFC используют одинаковые типы данных для одних и тех же сущностей
- [ ] Интерфейсы между RFC совместимы (producer/consumer событий совпадают)
- [ ] Нет дублирующихся RFC для одного и того же компонента
- [ ] Зависимости RFC (depends_on) корректны и нет циклов
- [ ] Для каждой пары RFC с зависимостью есть интеграционный тест-файл (`tests/integration/rfc-N-rfc-M.py`) — или явно отмечено "не применимо"

## 5. MEMORY.md актуальность

- [ ] Все принятые решения из PRD отражены в MEMORY.md
- [ ] Все принятые решения из ADR отражены в MEMORY.md
- [ ] Нет противоречий внутри MEMORY.md
- [ ] Нет "решённых" вопросов которые противоречат документам

## 6. Acceptance Criteria (только RFC)

- [ ] Каждый RFC имеет секцию Acceptance Criteria
- [ ] Каждый критерий конкретен и проверяем
- [ ] Критерии покрывают happy path и основные error cases

## 7. Нерешённые вопросы

- [ ] Все нерешённые вопросы из документов собраны в JOURNAL.md
- [ ] Каждый вопрос имеет владельца или трекинг

## 8. Безопасность (только RFC)

- [ ] Каждый RFC имеет заполненную секцию "Безопасность" (не пустые заглушки)
- [ ] Авторизация описана для всех публичных endpoint'ов / функций
- [ ] Пользовательский ввод: есть явная валидация или указано "нет пользовательского ввода"
- [ ] PII и секреты не упомянуты в секции "Наблюдаемость" как логируемые

## 9. Наблюдаемость (только RFC)

- [ ] Каждый RFC имеет заполненную секцию "Наблюдаемость"
- [ ] SLO конкретные (p99 ≤ X ms, error rate ≤ Y%) — не абстрактные
- [ ] Метрики определены с labels/tags

---

## Формат отчёта

Пиши в `.i2c/scratch/consistency-report.md`:

```markdown
# Consistency Report — [дата]

## Summary
✅ [N] пунктов OK
⚠️ [N] предупреждений
❌ [N] критических проблем

## Критические проблемы (требуют немедленного исправления)
- ❌ [Проблема]: [RFC-X утверждает Y, ADR-Z принял Z]

## Предупреждения
- ⚠️ [Предупреждение]: [RFC-X не имеет Acceptance Criteria]

## Рекомендации
- [Что сделать в первую очередь]
```
