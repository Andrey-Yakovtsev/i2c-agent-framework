# Протокол: Secure Code

**Активация:** только если RFC содержит секцию "Безопасность" или "Security". Если такой секции нет — этот протокол не применяется, пропусти полностью.

## 5 правил Secure by Default

### 1. Валидация входов на границе

- Все внешние входы (HTTP-параметры, CLI-аргументы, содержимое файлов) — валидируй в точке входа
- **Whitelist > blacklist:** определи что ДОПУСТИМО, отклони всё остальное
- Проверяй: тип, длину, формат, диапазон
- Невалидный вход → немедленный отказ с RFC error code

### 2. SQL: только параметризованные запросы

- Никогда не конкатенируй/интерполируй строки в SQL
- Используй параметризованные запросы или query builder ORM
- Если RFC требует raw SQL — используй плейсхолдеры (`$1`, `?`, `%s` в зависимости от стека)

### 3. Секреты только из env

- API-ключи, пароли, токены — только из переменных окружения
- Никогда в исходном коде, конфигурационных файлах или логах
- Убедись что `.env` в `.gitignore`

### 4. Auth на каждом эндпоинте

- Каждый endpoint/handler проверяет аутентификацию и авторизацию
- Не полагайся на фронтенд для контроля доступа
- Если RFC определяет роли/permissions — enforce server-side

### 5. Зависимости из lock-файла

- Устанавливай из lock-файла (`pip install -r requirements.txt`, `npm ci`, `go mod download`)
- Не добавляй unpinned зависимости
- После изменения зависимостей — запусти security-линтер (таблица в `protocols/code-quality.md`, колонка Security)

## Observability Security

- **Никогда** не логируй secrets, токены, API-ключи (даже частично)
- **Никогда** не логируй PII (email, телефон, имя) на уровне INFO; DEBUG — только если RFC явно требует
- **Маскируй** sensitive fields в error responses — показывай error code, не внутренние детали
- **Audit log** для auth-событий (login, permission denied, token refresh) — если RFC определяет такие события
