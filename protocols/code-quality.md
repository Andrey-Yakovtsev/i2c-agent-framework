# Протокол: Code Quality

Обязательные проверки для coding-агентов. Стек читай из MEMORY.md.

## Структура кода

- Один модуль = одна ответственность из Implementation Plan
- Публичный API модуля = только интерфейсы из RFC; всё остальное — private/internal
- Файл > 300 строк → разбей по ответственности

## Layered Architecture

Проверь свой код по чеклисту:

- [ ] **Transport/handler** (routes, controllers) содержит только: parse request → call service → format response. Бизнес-логики — ноль
- [ ] **Бизнес-логика** не импортирует фреймворк (без Flask/Express/Gin/FastAPI в service-слое)
- [ ] **Доступ к данным** за интерфейсом (repository pattern) — бизнес-логика вызывает интерфейс, не ORM/SQL напрямую
- [ ] **Конфигурация** — отдельный слой; в бизнес-логике нет захардкоженных URL, credentials, feature flags

## 12-Factor (code-relevant)

Только пункты, которые ты контролируешь при написании кода:

- [ ] **Конфиг** через env vars (`os.environ` / `process.env` / `os.Getenv`), не хардкод
- [ ] **Зависимости** явно объявлены (requirements.txt / package.json / go.mod) — не используй необъявленные импорты
- [ ] **Логи** в stdout (`print` / `console.log` / `log.Println`), не в файл
- [ ] **Stateless** — нет in-memory состояния между запросами; всё состояние в external store
- [ ] **Порт** из env var, не захардкожен

## Обработка ошибок

- **Валидируй на границе:** все внешние входы проверяй в точке входа (тип, диапазон, формат)
- **Fail fast:** валидация не прошла → сразу возврат с RFC error code
- **Нет fallback в RFC → пробрасывай:** если RFC не описывает recovery, не выдумывай свой
- **Коды ошибок дословно из RFC:** точные коды и сообщения из error-таблицы
- **Нет голых except/catch:** всегда указывай тип ошибки

## Линтеры и форматтеры

Определи стек из MEMORY.md, примени соответствующую строку:

| Стек | Формат | Линт | Security |
|------|--------|------|----------|
| Python | `ruff format .` | `ruff check --fix .` | `bandit -r .` |
| Node/TS | `prettier --write .` | `eslint --fix .` | `npm audit --audit-level=high` |
| Go | `gofmt -w .` | `golangci-lint run` | `gosec ./...` |

Стек не в таблице → используй стандартный форматтер/линтер языка. Если в MEMORY.md указан конкретный инструмент — используй его.

## Финализация

Последовательность:
1. **Format** — запусти форматтер
2. **Lint + fix** — запусти линтер с автофиксом
3. **Re-check** — перепроверь линтер (должен пройти без ошибок)
4. **Security lint** — если активен `protocols/secure-code.md`: HIGH/CRITICAL → исправь, warnings → запиши в отчёт

Запиши итог в секцию "Результат линтеров" модульного отчёта.
